# Phase 1, Task 2 — Make orchestration + GNU make guard

## Context
Make is the single task entrypoint across the monorepo. Sub-project Makefiles will eventually be regenerated by `@sdlcforge/gen-make`, so target naming must be canonical from day one.

## Acceptance
Root `users-module/Makefile` provides:
- GNU make version guard at the top: detect BSD make (macOS default `make` is 3.81; even GNU 3.81 on macOS is fine — we need to fail on `bmake`/non-GNU). Print explicit install instructions (`brew install make` and use `gmake`) and exit 1.
- Default target `build` → builds api and gui.
- `test`, `test.unit`, `test.integration` (delegate per sub-project).
- `lint`, `lint-fix`.
- `dev.start` → bring up `deploy/local/docker-compose.yml`, then run api + gui in foreground (use `make -j` or `parallel` pattern; document choice).
- `dev.stop` → tear down docker-compose.
- `clean` → remove build artifacts; preserve generated SQL/code.
- All targets must be `.PHONY`.
- Targets that delegate to sub-projects use dot notation: `build.api`, `build.gui`, `test.api`, `test.gui`, etc.

Sub-project Makefiles in `model/`, `api/`, `gui/` each:
- Implement the same canonical contract for that sub-project.
- May be hand-written for now; mark with header comment `# REGENERATED BY @sdlcforge/gen-make — do not edit directly` so future regeneration is signaled.

## Out of scope
docker-compose contents (Task 1.3) and api config/OTel (Task 1.4).

## How to verify
- `make help` (add a self-documenting help target) lists all targets with brief descriptions.
- `make build` succeeds against the placeholder api + gui from Task 1.1.
- On a system with non-GNU make, `make` exits 1 with the install message.
